# v0.2 diff scorer 优化总结（Week 6 → Week 11）

## 结论先行

从 Week 6 到 Week 11，一共探索了 6 条 lever 来修掉 arXiv 1706 在 semantic oracle
下那几条"短 token 重复 + 段落搬家"的 relocated-vs-broken 误判，没有一条做到
**"default-on 在所有 benchmark 上都不回退 + arXiv 真正修对"**。

**决定**：

- **v0.2 diff scorer 冻结在 v0.2.1 的默认行为上**。
- 所有 Week 9 / Week 10 / Week 11 的新机制（broken floor、ctx concat mode、ctx-aware
  preemption）都只作为 **opt-in 环境变量** 保留在代码里，默认关闭。
- **不 tag v0.2.2**。没有用户可感知的 default 改进，tag 一个版本号会误导使用者以为
  精度变了。
- benchmark baseline 继续以 `benchmarks/baselines/v0.2.1.json` 为准，`compare_baseline`
  regression guard 不放松。
- 转产品节奏：diff workflow 的用户侧可用性（文档、示例、review TUI）比再压 1-2 个
  case 更有价值。

---

## 当前 baseline（v0.2.1 default，两种 oracle）

| benchmark | semantic(1to1) | old(rank-k) |
|---|---|---|
| arXiv 1706.03762 | 84.6% / 6 fail (loc 76.5%) | 92.3% / 11 fail (loc 56.4%) |
| BERT 1810.04805 | 92.9% / 1 fail | 100.0% / 0 fail |
| Word2Vec 1301.3781 (stressed) | 80.0% / 3 fail (loc 100%) | 86.7% / 4 fail (loc 57.1%) |
| Seq2Seq 1409.3215 | 100.0% / 0 fail | 100.0% / 0 fail |
| revised (small synthetic) | 88.5% / 3 fail | 88.5% / 3 fail |

arXiv 的 6 个 semantic failure 是这轮调优的主要目标。它们共性是：

- 短 token 在一个 page 内重复 ≥ 3 次（"Multi-Head Attention"、"Scaled Dot-Product
  Attention"、"BLEU"）
- v2 里段落整体移位，但同 token 的位置排名（rank）变化
- scorer 本应返回 broken 或指向 semantic 正确的那次 occurrence，实际指到另一个
  occurrence —— `pred=relocated` 但 `gt=broken`

## 每条 lever 的结论

### 1. Global Hungarian allocator（Week 2 B，已 revert）

全局最优 1:1 匹配替 greedy。arXiv `anc_68ab8fb0 Multi-Head Attention` 这条 case
天生不能靠全局 sum-max 解决 —— "牺牲它换两个高分 anchor 各得其所" 反而推翻了本来
greedy 给出的正确结果。回退。

**验收**：失败。`pdfanno/diff/match.py` 不保留 Hungarian 代码。

### 2. section-aware layout subscore（Week 3 C，已保留）

section_sim 作为 layout_score 的子分（权重 0.20），帮 "段落名一致" 的 case。保留
在 default。`PDFANNO_DISABLE_SECTION_SIM=1` 可关掉做反事实。

**验收**：通过。默认行为。

### 3. per-anchor ctx window（Week 4，已 revert）

`_context_window` 默认把 `norm_page.find(selected)` 返回的是 page 里第一次出现的
位置 —— 同 page 同 token 的多个 anchor 共享同一段 ctx 字符串。Week 4 尝试改成
"按 anchor 自身 quad 位置取 ctx"，结果 arXiv 92.3%→89.7% / loc 56.4%→48.7% 回退。
回退。

**教训（load-bearing constraint）**：**"first-match ctx anchoring" 是设计约束**。
它让同 token anchor 看不到细节差异（损失），但它让 context 字串从 page 级定锚、
从而让 rank/layout 子分数能主导整体 score（收益）。整体更稳。

这个约束直接决定了后面 Week 9 / 11 的上限。

### 4. Hungarian within same-token group（Week 7，已 revert）

保持组间 greedy，但同 token 组内做 sum-max。2/3 的 benchmark 零变化（因为每组一般
只有 1 个 anchor 或已经匹配正确），arXiv 依然不过关（`anc_68ab8fb0` 照常错）。
无净收益，回退。

**教训**：用 "局部 assignment 优化" 救 "对 anchor 感知过弱的 score" 是药不对症。

### 5. broken floor（Week 9，保留为 opt-in）

当 `_classify` 要返回 relocated 但 `ctx_sim < floor` 时强制返回 broken。same-location
preserved 不受惩罚。

阈值 sweep 看到：

- W2V semantic 3→1（floor=0.02）或 3→0（floor=0.10），**有用**
- arXiv semantic 任意 floor 都 ≥6 failure，甚至在 floor=0.10 时 8 failure
- old oracle 全线 −13pp 左右

**验收**：单 benchmark 有效但不能做 default。改为 opt-in。

开关：
```
PDFANNO_BROKEN_CTX_FLOOR=0.10       # 启用并指定阈值
PDFANNO_DISABLE_BROKEN_FLOOR=1      # 显式强关
```
默认值常量 `BROKEN_CTX_FLOOR=0.0`（= 关闭）。

### 6. ctx concat mode（Week 10，保留为 opt-in）

`_context_similarity` 有两种算法：`mean`（v0.2.1 原始行为，before/after 各一次
`SequenceMatcher.ratio()` 取均值）和 `concat`（和 semantic oracle 一致，before||after
拼一次 `SequenceMatcher`）。抽到 `pdfanno/diff/context.py` 共享 helper。

配 broken floor 在 concat+0.10 时首次让 arXiv semantic 6→4：

| mode / floor | arXiv sem | W2V sem |
|---|---|---|
| mean / 0.00 (v0.2.1) | 84.6% / 6 | 80.0% / 3 |
| mean / 0.10 | 79.5% / 8 | **100% / 0** |
| concat / 0.10 | **89.7% / 4** | 86.7% / 2 |

**验收**：arXiv 第一次真收益，但同时把 W2V 从 3→0 变回 3→2。没有单一 (mode, floor)
对所有 benchmark 净正。改为 opt-in。

开关：
```
PDFANNO_CTX_SIM_MODE=concat         # 切换到 concat
PDFANNO_CTX_SIM_MODE=mean           # 或不设为默认
```
oracle 端永远用 concat（那是 oracle 的定义，不走环境变量）。

### 7. ctx-aware assignment preemption（Week 11，保留为 opt-in）

greedy `_assign_one_to_one` 里加一条可选 preemption 规则：candidate slot 已被
holder 占用时，若 contender 是 same-token + score 接近（ε=0.05）+ ctx 显著优势
（Δ=0.10），contender 抢占，holder 进 displaced 队列后找次优。

跑 A/B 发现 arXiv 32 个 same-token slot 冲突里 **30 个 `ctx_delta = 0`**——同 token
anchor 因为 Week 4 的 `_context_window` first-match anchoring，对同一 candidate
看到的 ctx 数值字面一致。规则永远没机会触发。唯一触发的一条（Scaled Dot-Product /
BLEU 那一对）把两个 anchor 都抢错，净负。

**验收**：机制对，信号太平，打不响。arXiv 净贡献 0。改为 opt-in。

开关：
```
PDFANNO_CTX_AWARE_ASSIGN=1          # 启用
PDFANNO_CTX_ASSIGN_EPSILON=0.05     # score 相近阈值
PDFANNO_CTX_ASSIGN_MIN_ADVANTAGE=0.10  # ctx 优势阈值
```

## 设计约束环（为什么 6 条 lever 全卡在同一堵墙）

```
   Week 4 first-match ctx anchoring
     ↓（副作用）
   同 page 同 token 的 anchor 共享 ctx 字串
     ↓
   Week 9 broken floor 和 Week 11 preemption 需要"ctx 在同 token 之间有区分度"
     ↓
   没有区分度 → lever 要么打不响、要么乱打
     ↓
   （想解决）改回 per-anchor ctx
     ↓
   Week 4 已证伪：整体 arXiv 回退

```

换句话说：**"同一 lever 无法同时满足 arXiv 的 rank/layout 依赖 + W2V 的 ctx-based
broken 判断"**。这不是 bug，是两个 benchmark 偏好不同的信号组合。

## 保留在代码里的 opt-in 开关清单

| 开关 | 默认 | 用途 |
|---|---|---|
| `PDFANNO_DISABLE_SECTION_SIM=1` | section_sim 启用 | 反事实：关掉 section 子分数看 layout 是否还立得住 |
| `PDFANNO_BROKEN_CTX_FLOOR=<float>` | `0.0`（关） | 启用 Week 9 broken floor 并指定阈值 |
| `PDFANNO_DISABLE_BROKEN_FLOOR=1` | 未设 | 强制关闭 broken floor（优先于数值） |
| `PDFANNO_CTX_SIM_MODE=mean\|concat` | `mean` | 切换 matcher 的 ctx similarity 算法 |
| `PDFANNO_CTX_AWARE_ASSIGN=1` | 未设 | 启用 Week 11 preemption |
| `PDFANNO_CTX_ASSIGN_EPSILON=<float>` | `0.05` | preemption 里 score 相近阈值 |
| `PDFANNO_CTX_ASSIGN_MIN_ADVANTAGE=<float>` | `0.10` | preemption 里 ctx 优势阈值 |

全部默认关闭或保持 v0.2.1 行为。`compare_baseline` 在 CI 下 default 环境跑，任何
default 改动都必须先过 regression。

## 仍未解决的硬 case

- arXiv `anc_68ab8fb0 Multi-Head Attention` 全局级 relocation 误判。short-token
  高重复 + 跨 section 搬家，Week 2/4/7/11 都没打下。
- arXiv `anc_bc48e155 Scaled Dot-Product Attention` 和 `anc_e3ba2da8 BLEU` 那一组
  "own best ctx 存在但被别人抢走，退到次优"。Week 11 preemption 是为它们设计的，
  但 ctx 信号太平没打响。
- revised 那 3 条 semantic failure（两种 oracle 一致），尺寸太小暂不深挖。

这些属于 **"已知 hard case，研究性保留"**。未来若引入 embedding-based ctx 或
per-anchor ctx 的混合方案，可再回来打。

## 后续研究方向（均延后，不进 v0.2）

1. **分层 ctx**：matcher 默认用 first-match ctx（保护 arXiv 稳），但 assignment /
   classify 层单独算一份 per-anchor ctx 只做局部决策。Week 4 的坑挖第二次，但只用
   在决策层、不污染主 score。
2. **ctx embedding**：用 sentence embedding 代 `SequenceMatcher.ratio()`。在同
   token 场景下语义相似度有可能给出 Week 11 需要的区分度。但基础设施代价大，
   v0.2 不做。
3. **fuzzy 候选 ctx 权重调整**：`text_score < 1.0` 时 ctx / layout 的相对权重是否
   应该动态化。目前固定权重是对 exact-match 优化的，fuzzy 段用它略亏。

## 工具沉淀（Week 6-11 副产品）

这轮调优留下几样对未来仍然有用的基础设施：

- `benchmarks/tools/ground_truth_semantic.py`—— 1-to-1 semantic oracle，消除了
  旧 rank-k-to-k oracle 在 v2 token 增删时的位移噪声。任何未来的 scorer 实验都应
  以它为主评估 oracle。
- `benchmarks/tools/compare_baseline.py`—— 多 benchmark regression guard。CI 跑
  default env 下的对比，任何 default 变更必须 baseline 全绿才能 merge。
- `benchmarks/baselines/gt_semantic_*.json`—— 三个大 benchmark 的 semantic GT，
  固定在 repo 里，不依赖重跑 oracle。
- `pdfanno/diff/context.py`—— matcher 和 oracle 共享的 ctx helper。以后任何 ctx
  相关实验（embedding、分层 ctx、mode 切换）都在这里加，不会再出现"两边说的不是
  同一种语言"。
- 7 个 env 开关—— 允许未来在不动 default 的前提下，单独跑 A/B sweep。

## 下一步（产品方向，脱离 scorer 调优）

离开 scorer 的 −1 case 游戏，转去做 **diff workflow 的用户侧可用性**。三条候选
按工作量从小到大：

1. **文档 + 示例**：给 `pdfanno diff` 写一份完整的 workflow guide（old.pdf + 新版
   → diff JSON → 人工审核 → 下一步）。把 preserved / relocated / changed / broken
   四态和 confidence score 的含义写清楚。
2. **真实 demo / case study**：拿一个公开 paper 的两版 PDF 跑 diff，把 JSON 输出、
   人工判断、最终 decision 做成示范。
3. **`pdfanno review diff.json` 交互式 TUI**：接管 diff 结果的人工审核环节。
   terminal 里一次一个 anchor，可看 v1/v2 两边 snippet，按 y 接受、n 拒绝、r
   标待审、q 退出。复用 Phase 2 TUI 基础设施的 30% 左右。

方向 3 在产品上价值最高 —— 没有它，diff 命令的输出还只是给 agent 消费的 JSON，
没有给人用的界面。但工作量也最大。
